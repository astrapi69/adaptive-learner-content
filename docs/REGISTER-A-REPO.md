# Register a content repo for the cross-repo search

Adaptive Learner can search **across several content repositories**, not
just the official one. To have your own repo included, you submit a
request, and that request is simply a **pull request against
[`recommended-repos.json`](../recommended-repos.json)** in this repo.

CI validates your repo at the exact commit you pin. Once it is green,
a maintainer merges it and your snapshot joins the federated search.

## Prerequisites

Your repo must be a valid Adaptive Learner content repo: it publishes a
`search-index.json` at its root (auto-generated by
`scripts/generate_search_index.py`, same as here) and its content passes
`scripts/validate_content.py`. See
[GETTING-STARTED.md](GETTING-STARTED.md).

## Steps

1. Pick the commit you want listed and copy its full 40-char SHA:

   ```bash
   git -C your-content-repo rev-parse main
   ```

2. Fork this repo and add an entry to `recommended-repos.json`:

   ```jsonc
   {
     "url": "https://github.com/your-name/your-content",
     "branch": "main",
     "commit": "<the 40-char SHA from step 1>",
     "title": "Your set collection",
     "description": "One line about what's inside",
     "trust_level": 1,
     "languages": ["de-fr"],
     "validation": {
       "status": "validated",
       "validated_at": "2026-07-09T00:00:00Z",
       "index_schema_version": "1.0"
     }
   }
   ```

   - `commit` is **required and pinned**: the search fetches your
     `search-index.json` at exactly this commit.
   - External repos start at `trust_level: 1`.
   - `validation.validated_at` is when your snapshot was checked; CI
     re-checks it on the PR regardless.

3. Open the PR. The **Validate registered repos** workflow will:
   - confirm the commit exists and is on the declared branch,
   - validate your `search-index.json` at that commit against
     [`schema/search-index.schema.json`](../schema/search-index.schema.json),
   - confirm the index's `repo` slug matches your URL.

4. Green + maintainer merge = your repo is in the search.

## Updating a listed repo

Pinning is strict on purpose: the search only ever serves a **validated
snapshot**. When you publish new content, open a **new PR** that bumps
your entry's `commit` (and `validated_at`) to the new SHA. It is
re-validated before it goes live; old, unvalidated commits are never
served.

## Validate locally first

```bash
pip install "jsonschema>=4,<5"
python scripts/validate_registry.py            # offline: file shape + rules
python scripts/validate_registered_repo.py \
    https://github.com/your-name/your-content   # online: your pinned commit
```
